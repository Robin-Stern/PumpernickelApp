Kontext: Wir haben aktuell eine FoodEntryScreen.kt, die zu viel Logik (Parsing, Validierung, State-Management) enthält. Wir möchten dies nach dem MVI-Prinzip (Model-View-Intent) refactoren, um die UI "dumm" zu machen und die Geschäftslogik im FoodViewModel zu zentralisieren. Das Ziel ist bessere Testbarkeit und saubere Trennung (Separation of Concerns).
Aufgabe:
1.
Erstelle einen UiState:
◦
Definiere im FoodViewModel (oder einer separaten Datei) eine Data Class FoodEntryUiState.
◦
Diese soll alle Felder des Screens enthalten (name, calories, protein, fat, carbs, sugar, barcode, isRecipe) sowie errorMessage und successMessage.
2.
Erstelle UI-Events (Intents):
◦
Definiere ein sealed interface FoodEntryEvent.
◦
Mögliche Events: OnNameChanged(String), OnCaloriesChanged(String), ..., OnSaveClicked, ClearMessages.
3.
Refactore das FoodViewModel:
◦
Implementiere einen MutableStateFlow<FoodEntryUiState>, den der Screen beobachtet.
◦
Implementiere eine Funktion onEvent(event: FoodEntryEvent), die den State entsprechend aktualisiert.
◦
Wichtig: Verschiebe die gesamte Validierungslogik aus dem Screen in das ViewModel.
◦
Parsing-Logik: Implementiere die Korrektur von Komma zu Punkt (.replace(',', '.')) direkt im ViewModel während des Speichervorgangs.
4.
Refactore den FoodEntryScreen:
◦
Entferne alle lokalen mutableStateOf-Variablen für die Formularfelder.
◦
Beobachte stattdessen den uiState vom ViewModel (via collectAsStateWithLifecycle oder collectAsState).
◦
Ersetze direkte Zuweisungen durch das Senden von Events, z.B.: onValueChange = { viewModel.onEvent(FoodEntryEvent.OnNameChanged(it)) }.
◦
Der "Speichern"-Button ruft nur viewModel.onEvent(FoodEntryEvent.OnSaveClicked) auf.
5.
Clean Code Check:
◦
Stelle sicher, dass der Screen keine toDoubleOrNull() Aufrufe mehr enthält.
◦
Achte darauf, dass die data class Food weiterhin nur mit validen Daten instanziiert wird (Nutze die vorhandenen require-Checks im init-Block).
