<instance_sharing_note>
**Single VM instance vs two separate instances:**

Both OverviewView and MainTabView call `GamificationUiKoinHelper().getGamificationViewModel()`. In Koin, `viewModel { ... }` (not `single { ... }`) is a factory that creates a new instance per `get()` call unless a ViewModelStoreOwner is provided. In iOS context there is no ViewModelStoreOwner, so we may get TWO distinct `GamificationViewModel` instances — one for the rank strip, one for the modal queue.

This is **acceptable** because:

- `rankState` is a `StateFlow<RankState>` derived from `gamificationRepository.rankState.stateIn(...)` — the upstream repository flow is a single `Flow<RankState>` from a Room query, so both VM instances eventually observe the same data (with a small memory overhead of two `stateIn` operators running). Visual parity is maintained.
- `unlockEvents` is a `SharedFlow<UnlockEvent>` derived directly from `gamificationEngine.unlockEvents` — the engine is a Koin `single { ... }` so there is ONE engine, ONE SharedFlow. Both VM instances forward the exact same SharedFlow, so the modal queue receives every event.

**Verification the executor should confirm:** when UnlockModalView fires, the rank strip XP changes within the same repository-flow cycle. This is covered by UAT re-run after execute.

**If a future phase demands shared instance semantics, the fix is to register `GamificationViewModel` as `single { ... }` in `GamificationUiModule.kt` instead of `viewModel { ... }` — but that is Kotlin-side work explicitly out of scope here per constraints.**
</instance_sharing_note>
