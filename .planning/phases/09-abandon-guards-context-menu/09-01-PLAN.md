Add `.confirmationDialog` modifier to the ScrollView inside `activeWorkoutView(_ active:)`, after the `.toolbar` modifier (per D-03, D-04):

```swift
.confirmationDialog(
    "Abandon Workout?",
    isPresented: $showAbandonDialog,
    titleVisibility: .visible
) {
    Button("Save & Exit") {
        viewModel.enterReview()
        viewModel.saveReviewedWorkout()
        dismiss()
    }
    Button("Discard", role: .destructive) {
        viewModel.discardWorkout()
        dismiss()
    }
    Button("Cancel", role: .cancel) { }
} message: {
    let completedSetsCount = exercises.reduce(0) { sum, ex in
        sum + ex.sets.filter { $0.isCompleted }.count
    }
    Text("Exercise \(exIdx + 1)/\(exercises.count), \(completedSetsCount) sets completed")
}
```

**Placement:** The `.confirmationDialog` must be attached inside the `activeWorkoutView` function body where `active`, `exercises`, and `exIdx` local variables are in scope (defined at lines 120-124). Attach it to the ScrollView chain, after `.toolbar { ... }`.

**Race condition note (from research Pitfall 1):** The sequential calls to `enterReview()` then `saveReviewedWorkout()` use `viewModelScope.launch` on `Dispatchers.Main.immediate` which is single-threaded. The first launch block completes before the second starts, so the state is guaranteed to be `Reviewing` when `saveReviewedWorkout()` checks. No dedicated `saveAndExit()` KMP method is needed.

**Dismiss timing (from research Pitfall 2):** `dismiss()` is called after firing both ViewModel methods. Since `viewModelScope` outlives the SwiftUI view, the coroutines complete even after the view is dismissed. Room writes are not interrupted.

**Destructive styling (Claude's discretion):** Using `Button("Discard", role: .destructive)` which automatically renders the text in red -- standard iOS pattern.

**Finish Workout goes straight to recap (Claude's discretion):** No additional confirmation needed -- firmware parity, `enterReview()` call from menu is direct.
