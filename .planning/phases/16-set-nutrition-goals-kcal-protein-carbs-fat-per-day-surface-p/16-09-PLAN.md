<objective>
Close VERIFICATION.md Gap 1 (CR-01 — iOS rings stale after editor save) and the two related warnings: WR-06 (iOS banner-flash on tab appearance) and IN-02 (Android double-refresh on first composition). Three small, surgical edits across two files. No new types, no API additions.

Purpose: After Plan 06 shipped the iOS editor sheet, an integration race remained — the editor saved goals to DataStore via its own factory-scoped OverviewViewModel instance, but OverviewView's separate VM instance held a stale `_uiState.nutritionGoals` because `refresh()` only runs on init. The user's manual workaround was tapping the toolbar refresh icon. This plan adds an `onDismiss:` callback so the parent view re-fetches automatically. The same OverviewView gets the banner-flash fix (default-false instead of default-true). Android gets a small dead-code cleanup. All three changes are independent of each other; bundling them keeps the gap-closure surface area minimal.
</objective>

<title>iOS: add onDismiss refresh callback to the editor sheet</title>
<read_first>

- iosApp/iosApp/Views/Overview/NutritionGoalsEditorView.swift (so the executor confirms the editor still uses its own VM and the parent simply re-fetches after dismiss)
</read_first>
<action>
In `iosApp/iosApp/Views/Overview/OverviewView.swift`, replace the existing sheet modifier block (lines 62-65):

```swift
.sheet(isPresented: $showEditor) {
    NutritionGoalsEditorView()
        .presentationDragIndicator(.visible)
}
```

with:

```swift
.sheet(isPresented: $showEditor, onDismiss: {
    viewModel.refresh()
}) {
    NutritionGoalsEditorView()
        .presentationDragIndicator(.visible)
}
```

This forces `OverviewViewModel.refresh()` to run after the editor sheet closes (whether closed by Save, by drag-down, or by tap-outside), which causes `_uiState` to re-fetch goals/macros/muscle-load and re-emit on `uiStateFlow`. `observeUiState()` already listens to that flow and writes the new values into `goals`, `todayMacros`, and `muscleLoad` — the rings then redraw with the saved goal values.

Do NOT change the editor's own ViewModel acquisition or pass a shared instance across the sheet boundary. The simpler `onDismiss` refresh is sufficient and matches Android's behavior where the same VM instance is shared via `koinViewModel()`/`ViewModelStore`.
</action>

<task id="09-02">
<title>iOS: change bannerVisible default to false (WR-06 banner-flash fix)</title>
<read_first>
- iosApp/iosApp/Views/Overview/OverviewView.swift (line 16 holds the @State declaration)
- iosApp/iosApp/Views/Overview/OverviewView.swift (lines 102-112 hold observeBannerVisible — confirm it sets bannerVisible from the StateFlow on every emission, including the first one, so default-false is corrected promptly)
</read_first>
<action>
In `iosApp/iosApp/Views/Overview/OverviewView.swift`, change line 16 from:

```swift
@State private var bannerVisible: Bool = true
```

to:

```swift
@State private var bannerVisible: Bool = false
```

Rationale: `observeBannerVisible()` (lines 102-112) writes `bannerVisible = v` for every emission of `viewModel.nutritionGoalsBannerVisibleFlow`. The flow's first emission carries the persisted DataStore value. With default-true, dismissed banners flash on every tab appearance for the brief window between tab-render and first-flow-emit. Default-false means the worst case is a one-frame "missing banner" for users who have NOT yet dismissed — far less visually distracting than the flash.
</action>

<action>
In `androidApp/src/androidMain/kotlin/com/pumpernickel/android/ui/screens/OverviewScreen.kt`, delete lines 88-90:

```kotlin
    LaunchedEffect(Unit) {
        viewModel.refresh()
    }
```

The block is dead code: `OverviewViewModel.init { refresh() }` already triggers a refresh on first VM construction. The LaunchedEffect spawns a second concurrent refresh on every composable re-entry, which wastes a coroutine and a DataStore read per refresh.

Do NOT remove the toolbar `IconButton(onClick = { viewModel.refresh() })` at lines 97-99 — that one is the intentional manual-refresh affordance.
</action>
