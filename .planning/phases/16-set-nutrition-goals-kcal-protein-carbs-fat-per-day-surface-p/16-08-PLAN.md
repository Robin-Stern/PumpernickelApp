<objective>
Close Gap 2 (WR-03 / WR-04) on both platforms: the editor's local field state must be
seeded ONCE from the first non-null DataStore emission and NEVER reset by subsequent
emissions. Today, Android uses `remember(storedStats)` / `remember(storedGoals)` as
state keys, so any re-emission of the source Flow rebuilds the local state, silently
discarding in-progress user edits. iOS's `observeStats()` and `observeGoals()` loops
unconditionally overwrite `@State` on every emission with the same effect.

The fix:
- Android: replace `remember(storedStats)` with `remember { ... }` (no key) plus a
  `LaunchedEffect(storedStats)` block that seeds fields only when a guard boolean
  `statsInitialized` is false. Same pattern for `storedGoals` / `goalsInitialized`.
  Guards are `rememberSaveable` so they survive configuration changes.
- iOS: introduce `@State private var statsInitialized: Bool = false` and
  `@State private var goalsInitialized: Bool = false`. Guard the `for try await`
  loops in `observeStats()` and `observeGoals()` so picker/text state is written
  ONLY on the first non-null emission.

This is a UI behaviour change. Human verification is required because the bug is a
race triggered by DataStore Flow re-emission timing — see VERIFICATION.md
human_verification tests #2 and #3.

Purpose: User edits in the editor never get silently overwritten.
Output: Editor screens on both platforms with one-shot initialization guards.
</objective>
