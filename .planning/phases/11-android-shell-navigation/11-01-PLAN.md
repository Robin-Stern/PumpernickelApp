    Create `androidApp/src/main/kotlin/com/pumpernickel/android/PumpernickelApplication.kt`:
    ```kotlin
    package com.pumpernickel.android

    import android.app.Application
    import com.pumpernickel.di.initKoin
    import org.koin.android.ext.koin.androidContext

    class PumpernickelApplication : Application() {
        override fun onCreate() {
            super.onCreate()
            initKoin {
                androidContext(this@PumpernickelApplication)
            }
        }
    }
    ```

    IMPORTANT: The shared `initKoin()` function signature is `fun initKoin(additionalModules: List<Module> = emptyList())`. It calls `startKoin { modules(...) }` internally. However, the `platformModule` on Android uses `androidContext()` which requires Koin's `androidContext()` to be called before modules are loaded.

    Check the actual `initKoin()` signature — if it does NOT accept a lambda for KoinApplication configuration, you will need to modify the shared module to support it. Two options:
    - Option A: Change `initKoin()` to accept a `KoinApplication.() -> Unit` config block
    - Option B: Call `startKoin` directly in PumpernickelApplication instead of using the shared helper

    Choose the approach that keeps iOS compatibility intact. If Option A, ensure the iOS caller in iosApp still compiles (iOS does not pass androidContext, it passes nothing or other config).

    This is the core navigation composable. Implementation:

    a) Define a sealed class or enum `TopLevelTab` with three entries:
    ```kotlin
    enum class TopLevelTab(
        val label: String,
        val icon: ImageVector,
        val route: Any
    ) {
        WORKOUT("Workout", Icons.Filled.FitnessCenter, WorkoutTabRoute),
        OVERVIEW("Overview", Icons.Filled.BarChart, OverviewTabRoute),
        NUTRITION("Nutrition", Icons.Filled.Restaurant, NutritionTabRoute)
    }
    ```

    b) Create the MainScreen composable:
    - State: `var selectedTab by rememberSaveable { mutableIntStateOf(0) }` (start on Workout tab, matching iOS)
    - Use `Scaffold` with `bottomBar` containing a `NavigationBar`:
      - For each `TopLevelTab`, add a `NavigationBarItem`:
        - `selected = index == selectedTab`
        - `onClick = { selectedTab = index }`
        - `icon = { Icon(tab.icon, contentDescription = tab.label) }`
        - `label = { Text(tab.label) }`
    - Scaffold content: Use a `Box` or simple conditional rendering based on `selectedTab`:
      - Tab 0 (Workout): Render a `NavHost` with `navController = rememberNavController()`, startDestination = `TemplateListRoute`:
        - `composable<TemplateListRoute> { WorkoutPlaceholderScreen() }` — placeholder for now
        - Future phases will add: TemplateEditorRoute, ExerciseCatalogRoute, ExerciseDetailRoute, CreateExerciseRoute, ExercisePickerRoute, WorkoutSessionRoute
      - Tab 1 (Overview): `PlaceholderScreen(icon = Icons.Filled.BarChart, title = "Overview", message = "Track your training progress and stats. Coming soon.")`
      - Tab 2 (Nutrition): `PlaceholderScreen(icon = Icons.Filled.Restaurant, title = "Nutrition", message = "Log meals and track your macros. Coming soon.")`

    c) Per-tab back stack preservation (per D-06):
    - Each tab's NavHost should use its own `NavHostController` created with `rememberNavController()` and retained across tab switches.
    - Use `key(selectedTab)` or store navControllers in a `remember` map, OR use the simpler pattern of rendering all three NavHosts in a Box with visibility toggling:
      ```kotlin
      val workoutNavController = rememberNavController()
      // ... other controllers
      Box {
          // Workout NavHost always composed, visible when selected
          AnimatedVisibility(visible = selectedTab == 0) {
              NavHost(navController = workoutNavController, startDestination = TemplateListRoute) { ... }
          }
          // etc.
      }
      ```
    - The critical requirement: switching from Workout to Overview and back must NOT reset the Workout tab's navigation stack.

    d) Material 3 NavigationBar colors:
    - The NavigationBar should use the default Material 3 colors, which will pick up the primary color from the theme for the selected indicator.
    - If the default indicator color does not match the green accent well enough, override with:
      ```kotlin
      NavigationBar(
          containerColor = MaterialTheme.colorScheme.surface,
      )
      ```
      and let NavigationBarItem use default colors (which use primary for selected state).
