<domain>
## Phase Boundary

A pure-presentation follow-up to Phase 15 that makes the gamification system **legible**. Phase 15 already awards XP, unlocks achievements, and assigns CSGO-style ranks — but the only visible surfaces are the Overview XP strip (one current tier) and transient unlock toasts. Users have no way to browse:

1. The **rank ladder** — all 10 tiers in order, which one they are on, which they have passed, each tier's XP threshold, and what is next.
2. The **achievement catalog** — every achievement (locked + unlocked), progress bars on partially-completed ones, unlock state + date, tier metadata (reward XP), category grouping.

This phase wraps existing domain/data artifacts (`GamificationRepository`, `RankLadder`, `Rank`, `RankState`, `AchievementCatalog`, `AchievementStateSeeder`) with new shared presentation + platform UI. **No new domain logic, no new XP sources, no schema changes.**

### What changes
- NEW shared ViewModel `RanksAndAchievementsViewModel` in `commonMain` exposing `rankLadderState: StateFlow<RankLadderUiState>` + reuses `AchievementGalleryViewModel` for the achievement half (D-21 already built it).
- NEW iOS KoinHelper (per Phase 15 convention) so Swift can obtain the new VM without touching `KoinPlatform.getKoin()` directly.
- NEW Android "Rank Ladder" screen + Compose composable.
- NEW iOS `RankLadderView.swift` + `RanksAndAchievementsView.swift` (container that combines ladder + achievements) written by the user per the MEMORY.md convention; this phase stops at the VM contract + KoinHelper factory and documents what the user should build.
- MODIFIED Overview entry: the `OverviewRankStrip` (Android + iOS) gains a tap affordance (or adjacent "View all" button) opening the browser. On Android the Overview tab currently has no NavHost — this phase must add one or promote the browser to a root-level destination.
- The existing Settings → Achievements entry (Android `SettingsSheet`, iOS `SettingsView`) may be consolidated with the new Overview entry or left untouched — planner decides per D-21 / D-18 balance (locked decision: keep Settings entry for now, add Overview entry on top; consolidation is deferred).

## Implementation Decisions

### Scope Boundaries (locked)

- **D-151-01 (Presentation only):** No new domain logic, no new XP sources, no schema changes, no migration. This phase reads from the existing `GamificationRepository` Flows and the static `RankLadder` / `AchievementCatalog` objects. If something requires a new domain concept, stop and surface it — it does not belong here.
- **D-151-02 (No decision contradictions):** This phase preserves Phase 15 decisions:
  - D-08 / D-09 rank ladder (10 tiers, exponential ×1.5, BASE_XP=500) — the ladder view renders these values, does not re-compute them.
  - D-10 permanent ranks — ladder shows passed tiers as "passed" (monotonic, no re-locking).
  - D-11 unranked-until-first-workout — ladder view must render a sensible empty / "start here" state for `RankState.Unranked`; no stale/flashy pretend-rank.
  - D-14 four categories × Bronze/Silver/Gold × 12 families = 36 tiles — the gallery already renders this grouping via `AchievementGalleryViewModel.tilesByCategory`; this phase does NOT alter that VM.
  - D-17 Bronze/Silver/Gold XP rewards (+25/+75/+200) — shown as static metadata on each tile.
  - D-18 Overview strip remains the primary "current rank" surface; the browser is the "full catalog" surface layered on top of it.
  - D-21 Achievements under Settings stays. Overview-strip tap is ADDITIVE, not a replacement.
- **D-151-03 (Reuse `AchievementGalleryViewModel`):** The achievement half of the browser is exactly what Phase 15-09 already delivered. Do not build a second achievement VM. The new Ranks screen and the existing Achievement gallery screen are separate navigation destinations; the browser container just links to both (or renders both in a single scroll / tabbed view — planner's discretion). This means the new shared VM owns ONLY rank-ladder state; achievements reuse the existing Koin binding + existing iOS `AchievementGalleryKoinHelper` + existing `AchievementGalleryView.swift` (iOS) / `AchievementGalleryScreen.kt` (Android).

### Shared VM Contract

- **D-151-04 (VM name + location):** `RanksAndAchievementsViewModel` lives in `shared/src/commonMain/kotlin/com/pumpernickel/presentation/gamification/RanksAndAchievementsViewModel.kt`. Matches the Phase 15 convention for gamification VMs (`GamificationViewModel.kt`, `AchievementGalleryViewModel.kt` in the same package).
  - *Naming rationale:* the VM is named for the container surface the user reaches from the Overview XP banner. Even though it only owns rank-ladder state today (achievements reuse `AchievementGalleryViewModel`), the name matches the screen and leaves room to absorb combined state later without another rename.
- **D-151-05 (Exposed state):**
  ```kotlin
  @NativeCoroutinesState
  val rankLadderState: StateFlow<RankLadderUiState>
  ```
  No separate `achievementCatalogState` — reuse `AchievementGalleryViewModel` for the achievement half.
- **D-151-06 (UI state model):**
  ```kotlin
  data class RankLadderUiState(
      val rows: List<RankRow> = emptyList(),
      val totalXp: Long = 0L,
      val currentRank: Rank? = null,         // null when Unranked
      val isUnranked: Boolean = true,
      val isLoading: Boolean = true
  )
  data class RankRow(
      val rank: Rank,              // tier 1..10
      val threshold: Long,         // RankLadder.thresholdFor(rank)
      val status: RankRowStatus,   // PASSED | CURRENT | LOCKED
      val xpToReach: Long?         // null for PASSED/CURRENT; threshold - totalXp for LOCKED
  )
  enum class RankRowStatus { PASSED, CURRENT, LOCKED }
  ```
  Pure data, Kotlin types only (no platform-specific types). `Rank` carries `displayName` and `tier` (1-based ordinal) — see `Rank.kt`.
- **D-151-07 (Derivation rules):** Row classification is a pure function of `rankState` + `totalXp`:
  - `RankState.Unranked` → all 10 rows marked `LOCKED` with `xpToReach = threshold` except SILVER which is `LOCKED` with `xpToReach = 0` and a hint that it unlocks on first workout (D-11 contract); `currentRank = null`, `isUnranked = true`.
  - `RankState.Ranked(currentRank, totalXp, ...)` → every row strictly below `currentRank.ordinal` is `PASSED`, the row at `currentRank.ordinal` is `CURRENT`, every row above is `LOCKED` with `xpToReach = max(0, threshold - totalXp)`.
  - Order rows by `Rank.values()` (Silver first → Global Elite last).
- **D-151-08 (Observability):** Backed by `combine(GamificationRepository.rankState, GamificationRepository.totalXp)` — both are already exposed as Flows by the repository. Use `stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), RankLadderUiState(isLoading = true))` to mirror the `AchievementGalleryViewModel` / `GamificationViewModel` pattern exactly.

### DI Wiring

- **D-151-09 (Koin registration):** Add the VM to `shared/src/commonMain/kotlin/com/pumpernickel/di/GamificationUiModule.kt` (NOT a new module). Phase 15's `GamificationUiModule` already owns `GamificationViewModel`; `RanksAndAchievementsViewModel` belongs to the same surface. This avoids module proliferation.
  ```kotlin
  viewModel { RanksAndAchievementsViewModel(get()) }   // takes GamificationRepository
  ```
- **D-151-10 (iOS KoinHelper):** NEW file `shared/src/iosMain/kotlin/com/pumpernickel/di/RanksAndAchievementsKoinHelper.kt` mirroring `AchievementGalleryKoinHelper.kt` exactly:
  ```kotlin
  class RanksAndAchievementsKoinHelper {
      fun getRanksAndAchievementsViewModel(): RanksAndAchievementsViewModel =
          KoinPlatform.getKoin().get()
  }
  ```
  No parameters, no caching, one getter per VM. Matches the Phase 15-08 convention `GamificationUiKoinHelper` and Phase 15-09 `AchievementGalleryKoinHelper`.

### Android UI

- **D-151-11 (New Android screen):** `androidApp/src/androidMain/kotlin/com/pumpernickel/android/ui/screens/RankLadderScreen.kt`. Compose + Material 3. Lists all 10 ranks. Each row shows:
  - Tier icon / badge (reuse the `tint(for: rank)` colour banding from `OverviewRankStrip.swift` — port the same five tint bands to Compose).
  - Rank `displayName` + tier number (`#1 Silver`, `#2 Silver Elite`, ...).
  - Threshold `"{threshold} XP"`.
  - Status chip / icon: `PASSED` (checkmark), `CURRENT` (filled badge, highlighted), `LOCKED` (lock icon + `"{xpToReach} XP to unlock"` subtitle when xpToReach > 0, or `"First workout unlocks"` for SILVER when unranked).
  - If `isUnranked`, show a header card "Unranked — complete a workout to unlock Silver" (the load-bearing D-11 copy, same literal string used in `OverviewRankStrip.kt`).
- **D-151-12 (Browser container):** On Android, add a single new destination `RanksAndAchievementsRoute` (`@Serializable data object`) hosted on the OVERVIEW tab (see D-151-15). Inside the screen: a simple scroll that embeds the rank ladder section on top and a "View all achievements" `ListItem` at the bottom that navigates to the existing `AchievementGalleryRoute` — **or** the planner may choose a M3 TabRow (Ranks / Achievements) if the nav plumbing is simpler. Decision is Claude's discretion BUT keep it one scrollable composable so the user's "View all" tap lands in a single legible surface, not a two-click drill-down.
- **D-151-13 (Route registration):** Add `RanksAndAchievementsRoute` to `androidApp/src/androidMain/kotlin/com/pumpernickel/android/ui/navigation/Routes.kt`. Wire a `composable<RanksAndAchievementsRoute>` into the Overview-tab NavHost (see D-151-15).
- **D-151-14 (Overview strip entry):** Modify `androidApp/src/androidMain/kotlin/com/pumpernickel/android/ui/screens/OverviewRankStrip.kt` to accept an `onTap: () -> Unit` callback and wrap the card in `Modifier.clickable(onClick = onTap)`. The `OverviewScreen` supplies `onTap = { navController.navigate(RanksAndAchievementsRoute) }`. iOS mirror via `onTapGesture { ... }` handled by the user.
- **D-151-15 (Overview tab needs a NavHost):** Today, Android's Overview tab in `MainScreen.kt` is a bare `OverviewScreen()` — there is no NavHost. This phase introduces `overviewNavController = rememberNavController()` + a NavHost with start destination `OverviewRootRoute` (`@Serializable data object`, renders the existing `OverviewScreen`) and `RanksAndAchievementsRoute`. Also re-add `AchievementGalleryRoute` to this Overview NavHost so "View all achievements" from the browser navigates within the Overview back stack (the existing `AchievementGalleryRoute` on the Workout tab's NavHost stays for the Settings-sheet entry per D-21). **The existing Settings → Achievements route stays on the Workout tab's `workoutNavController` untouched.**

### Metadata Display

- **D-151-18 (Rank row XP label):** `"{threshold:,} XP"` (locale-aware thousand separator on the UI layer, not in the VM). Use Compose `"%,d".format(threshold)` on Android and SwiftUI `NumberFormatter.localizedString(from:number:)` on iOS.

Keep it a small secondary-text line, not a shout. This is the one intentional touch on Phase 15-09's output; out-of-scope otherwise.
- **D-151-20 (Progress bar on partially-completed achievements):** Already present in the existing gallery via `currentProgress / threshold`. No changes required unless the planner finds a visible gap during planning (verify first).

SEED-001 captures the gap; addressing it is a separate backlog item.

<specifics>
## Specific Ideas

- The user's Overview XP banner tap → opens browser. Concretely: when rank is `Ranked`, the whole strip card is tappable; when `Unranked`, the tap still opens the browser so the user can see what ranks exist. Make the whole card the tap target, not just a "View all" affordance — one target is clearer than two.
- The browser container should feel like "everything about the gamification system" in one place. Ranks on top, achievements below (or a M3 tab switch). The user went from "I see rank X" + "toast says I unlocked Y" to "I can see the whole ladder and the whole catalog" — that's the legibility win.
- Keep the existing Settings → Achievements entry. Removing it is a separate UX call; consolidation is not in scope for this phase. Two entry points into the same destination is fine (Android: Settings route is on Workout tab's nav controller; Overview entry is on Overview tab's new NavHost — back-stack separation is the right mental model).
- Reward-XP label on achievement tiles (D-151-19) is a small but visible upgrade that makes "why does this matter" obvious on the gallery — every tile now says "Reward: +75 XP" etc. in secondary text.
- When rendering the ladder in Unranked state, SILVER should NOT say "500 XP to unlock" (threshold is 0). It should say "First workout unlocks" — that's the D-11 contract surfaced at row granularity.

</specifics>
